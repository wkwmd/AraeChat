### BareMetal v1.0 공식 최종 명세서

#### 1. 철학 (Philosophy)

1.1. BareMetal v1.0은  
- **“거꾸로 차감하여 가능성을 0으로 만들고, 앞으로 덧셈으로만 상태를 확정한다”**  
  는 원칙을 따른다.

1.2. 이 원칙에 따라,  
- OS/언어/플랫폼/구현에 따라 결과가 달라질 수 있는 여지는 **모두 금지하거나 상수로 고정**하고,  
- 허용된 선택지는 **단 하나의 결정론적 실행 경로**만 남도록 명세를 닫는다(seal).

---

#### 2. 스코프 및 Out-of-Scope

2.1. v1.0은 **단일 프로세스 / 단일 워크스페이스**를 전제로 한다.

2.2. 다음 항목들은 v1.0의 **명시적 스코프 밖(Out of Scope)** 으로 한다.  
이들은 어떤 방식으로도 v1.0의 합격/불합격 판정에 영향을 주지 않는다.

- 멀티스레드 및 동시성 제어.
- 여러 워크스페이스 간 상호작용 및 경쟁 조건(TOCTOU).
- 전원 차단 / OS 크래시 / 디스크 내구성 보장 수준(`fsync` 등)과 관련된 영속성 의미.
- 네트워크 I/O 및 외부 시스템 연동.

2.3. v1.0은 다음 OS 조합만을 **지원 스코프**로 명시하고, 이외 OS는 v1.0 스코프 밖이다.

- Windows
- Linux
- macOS

---

#### 3. 전역 금지 규칙 (Forbidden)

3.1. v1.0에서는 다음을 **전역 금지**한다.

- **외부 엔트로피 의존**  
  - 시스템 시간, 고해상도 타이머, OS 난수, 환경 변수, 사용자 입력 등 외부 상태에 따른 값 사용.
- **부동소수점 연산**  
  - float, double 등 부동소수점 타입 및 관련 연산 전면 금지.
- **플랫폼/컴파일러 의존 구조체 패킹 및 정렬(alignment)**  
  - 구조체를 곧바로 바이너리에 덤프하는 식의 불명확한 직렬화 금지.
- **외부 셸 스크립트 및 해시 도구 의존**  
  - CRC/해시 계산을 외부 명령어 또는 툴에 위임하는 행위 금지.
- **OS별 실행 분기 로직**  
  - 실행 경로가 OS에 따라 분기되는 로직(예: `if windows { ... } else { ... }`)을 통해 결과가 달라지는 것 금지.

---

#### 4. 파일 및 파일시스템 규칙

4.1. **파일 이름 및 개수**

- v1.0에서는 워크스페이스 내 **최종 결과 파일**을  
  - 정확히 한 개, 이름은 **`snaps.bin`** 으로 고정한다.
- 임시 작업 파일은  
  - 정확히 한 개, 이름은 **`snaps.tmp`** 으로 고정한다.
- 그 외 `.bin` 산출물 생성은 v1.0에서 허용하지 않는다.

4.2. **임시 파일 위치**

- `snaps.tmp`는 항상 `snaps.bin`과 **동일 디렉터리**에 생성되어야 한다.
- 다른 디렉터리에서 생성 후 이동하는 패턴은 v1.0에서 허용하지 않는다.

4.3. **덮어쓰기 금지 및 커밋 규칙**

- `snaps.bin`이 **이미 존재할 경우**:
  - 어떠한 경우에도 새 데이터를 쓰려고 시도해서는 안 되며,
  - 즉시 실패 상태로 종료되어야 한다.
- 새 스냅샷을 생성할 때:
  1. `snaps.tmp`에 전체 내용을 기록하고,
  2. 구조/CRC 검증을 모두 통과한 뒤에만,
  3. 파일 시스템의 `rename` 연산 1회로 `snaps.tmp` → `snaps.bin` 커밋을 수행한다.
- `rename` 직전, `snaps.bin`이 새로 생성되어 있지 않은지 **다시 확인**해야 하며,
  - 존재한다면 커밋을 중단하고 실패 상태로 처리해야 한다.
- 커밋 이후의 논리적 상태:
  - `snaps.bin`은 존재하고,
  - `snaps.tmp`는 존재하지 않는 상태가 되어야 한다(파일 시스템 특성을 고려한 논리 모델 기준).

---

#### 5. 파일 포맷 (바이트 레이아웃)

5.1. **전체 레이아웃**

`snaps.bin` (및 커밋 전의 `snaps.tmp`)의 전체 바이트열은 다음 순서를 갖는다.

1. Header (4바이트)
2. Term 시퀀스 바이트열 (고정 개수의 Term × 각 16바이트)
3. CRC32 (4바이트)

5.2. **Header (4바이트) – 절대 바이트열**

- 파일의 첫 4바이트(오프셋 0~3)는 아래와 같이 고정한다.

| 오프셋 | 값 (16진수) | 의미           |
|--------|-------------|----------------|
| 0      | `0x4D`      | ASCII 'M'      |
| 1      | `0x42`      | ASCII 'B'      |
| 2      | `0x00`      | 예약/버전 LSB  |
| 3      | `0x01`      | 버전 MSB (v1)  |

- 이 바이트열 자체가 스펙이며,  
  이를 어떤 정수로 해석할지는 구현 자유이나 결과 바이트열은 **반드시 위와 같아야 한다.**

5.3. **Term 시퀀스**

- Term의 정의 및 개수는 아래 6, 7장에서 별도 정의한다.

5.4. **CRC32 위치 및 범위**

- 파일의 **마지막 4바이트**(전체 길이 L이라 할 때, 오프셋 `L-4` ~ `L-1`)는 CRC32 값이 저장되는 영역이다.
- CRC32 계산 대상 바이트열은:  
  - **Header 4바이트 + 모든 Term 바이트열 전체**  
  - 즉, 마지막 CRC32 4바이트는 **계산 대상에서 제외**된다.

#### 6. Tick 및 Term 개수 규칙

6.1. **snapshot_tick 상수**

- v1.0에서 `snapshot_tick`은 아래의 단일 상수로 고정한다.
  - `snapshot_tick = 1023`

6.2. **tick 범위 및 순서**

- `tick`은 **정수**이며, 다음 범위를 가진다.
  - 수학적 정의:  
    - `tick ∈ { 0, 1, 2, ..., 1023 }`
- 즉,  
  - 최소 tick 값: 0  
  - 최대 tick 값: 1023  
  - 총 개수: `1023 - 0 + 1 = 1024`

6.3. **Term 개수**

- 파일에 기록되는 Term의 총 개수는 **정확히 1024개**로 고정한다.
- 누락되거나 초과되는 Term 개수는 v1.0에서 허용하지 않는다.

---

#### 7. Term 구조 및 직렬화 규칙

7.1. **Term 필드 구성**

각 Term은 다음 3개 필드로만 구성된다. 순서와 타입은 v1.0에서 **변경 불가**한 종료선이다.

1. `tick`  
   - 타입: 32비트 부호 없는 정수
2. `payload_a`  
   - 타입: 32비트 부호 없는 정수  
   - PRNG에서 파생
3. `payload_b`  
   - 타입: 64비트 부호 없는 정수  
   - PRNG에서 파생

7.2. **Term 크기**

- 각 Term의 총 크기는 **16바이트**로 고정된다.
- 패딩/패킹/정렬을 통한 추가 바이트는 **삽입해서는 안 된다.**

7.3. **Term 직렬화 (LE, 바이트 단위)**

각 Term은 다음 순서로 **리틀 엔디언(LE)** 규칙에 따라 직렬화된다.

1. `tick` → 4바이트 LE  
2. `payload_a` → 4바이트 LE  
3. `payload_b` → 8바이트 LE

여기서 LE란,  
- n바이트 정수 값의 최하위 바이트가 더 낮은 오프셋에 기록되는 규칙을 의미한다.

---

#### 8. PRNG (LCG) 규칙

8.1. **알고리즘**

- v1.0에서는 **선형 합동 생성기(Linear Congruential Generator, LCG)** 1종만 허용한다.
- 상태 전이 수식:

\[
X_{n+1} = (A \cdot X_n + C) \bmod M
\]

8.2. **파라미터 고정**

- v1.0에서 사용하는 파라미터는 아래의 단일 조합으로 고정한다.

- 모듈러:  
  - `M = 2^32`
- 곱셈 계수:  
  - `A = 1664525`
- 증가값:  
  - `C = 1013904223`
- 초기 시드(초기 상태):  
  - `X_0 = 42`

8.3. **출력 값**

- 각 step에서의 출력 PRNG 값은  
  - 상태 갱신 후의 값 `X_n`을 **하위 32비트 부호 없는 정수**로 해석한 것과 동일한 값으로 본다.
- 모든 연산은 정수 연산만 사용하며, 부동소수점 변환을 포함해서는 안 된다.

8.4. **tick 당 PRNG 호출 패턴 (payload 파생 규칙)**

- 각 `tick` 값에 대해, PRNG는 **정확히 3회** 호출되어야 한다.
- 순서 및 할당 규칙:

  1. 첫 번째 PRNG 호출 결과 → `payload_a` 전체  
  2. 두 번째 PRNG 호출 결과 → `payload_b`의 하위 32비트  
  3. 세 번째 PRNG 호출 결과 → `payload_b`의 상위 32비트  

- 수학적 결합:

\[
payload\_b = (PRNG\_3 \ll 32) + PRNG\_2
\]

여기서  
- `PRNG_2`, `PRNG_3`는 각각 두 번째, 세 번째 호출 결과(32비트 정수)이다.  
- `<< 32`는 정수의 비트 시프트 연산(상위 32비트로 이동)을 의미한다.

- 이 3회 호출 및 결합 외의 방식(예: 1회 호출 후 0 확장, 상태를 64비트 그대로 쓰기 등)은 v1.0에서는 허용하지 않는다.

---

#### 9. CRC32 규칙

9.1. **CRC32 파라미터 고정**

v1.0에서는 다음 파라미터 조합을 갖는 CRC32만 허용한다.

- 다항식(Polynomial): `0x04C11DB7`
- 초기값(Init): `0xFFFF_FFFF`
- 입력 비트 반전(RefIn): true
- 출력 비트 반전(RefOut): true
- 최종 XOR 값(XorOut): `0xFFFF_FFFF`

9.2. **입력 범위**

- CRC32는 다음 바이트열에 대해 계산된다.

\[
\text{data} = \text{Header(4B)} \;+\; \text{모든 Term 바이트열}
\]

- 마지막 CRC32 4바이트는 계산 대상에 포함되지 않는다.

9.3. **저장 위치**

- 계산된 CRC32 값은  
  - 파일 끝 4바이트에 **리틀 엔디언(LE)** 으로 저장된다.

---

#### 10. 검증 절차 규칙 (O(1) / O(n) 분리)

10.1. **O(1) 구조/크기 검증**

- 구조/크기 검증은 다음과 같이 **O(1)** 시간 안에 수행되어야 한다.

  - 파일 전체 길이가 최소한  
    - `Header(4B) + Term 1개(16B) + CRC(4B)` 이상인지 확인.
  - 파일의 첫 4바이트를 읽어,  
    - `[0x4D, 0x42, 0x00, 0x01]` 과 일치하는지 확인.

- 이 단계에서는 전체 파일을 읽지 않으며,  
  - 기본 구조가 성립하는지만 판단한다.

10.2. **O(n) CRC 검증**

- CRC 검증은 **O(n)** 시간 복잡도를 가진 별도 단계로 수행한다.
  - 전체 파일을 읽어들인 뒤,
  - 마지막 4바이트를 “저장된 CRC32”로 분리하고,
  - 나머지 부분(헤더 + 모든 Term 바이트열)에 대해 CRC32를 다시 계산한다.
  - 두 값이 일치해야만 유효한 스냅샷으로 인정한다.

- 구조/크기 검증(O(1))과 CRC 검증(O(n))은 **논리적으로 분리된 단계**로서 명세된다.

---

#### 11. 실행 절차 (Run) 및 Exit Code SSOT

11.1. **실행 절차 (Run)**

사용자는 같은 워크스페이스에서 다음과 같이 실행한다고 가정한다.

- 빌드:
  - 예: `cargo build --locked` (구현 문서에서 다룸)
- 실행:
  - 예: `cargo run -p bm-suite` (구현 문서에서 다룸)

명세 관점에서 중요한 것은,  
- “단일 실행 엔트리 포인트”가 위 4~10장의 규칙을 따르는 **전체 절차를 1회 수행**한다는 점이다.

11.2. **Exit Code SSOT**

- v1.0에서 관찰 가능한 종료 코드는 다음 두 값만 허용한다.

| Exit Code | 의미                                      |
|-----------|-------------------------------------------|
| 0         | PASS (정상 종료, 단일 합격 상태 달성)    |
| 1         | FAIL (모든 형태의 실패/오류/Hard Error)  |

- 2 이상의 Exit Code는 v1.0에서 사용하지 않으며,  
  합격/불합격의 SSOT(단일 진실 공급원)는 오직  
  **프로세스의 Exit Code 0/1** 만이다.

---

#### 12. v1.0 단일 합격 상태 정의

12.1. **1회차 실행 (클린 워크스페이스)**

- 워크스페이스에 `snaps.bin`이 존재하지 않는 상태에서 실행하면:

  1. `snaps.tmp`에  
     - 고정 Header 4바이트,  
     - `tick = 0..1023`에 대한 1024개 Term(각 16바이트, 규칙에 따른 PRNG 파생 payload),  
     - 위 바이트열에 대한 CRC32(규칙 9장)  
     가 정확히 기록된다.

  2. 구조/크기 검증(O(1))을 통과한다.
  3. CRC 검증(O(n))을 통과한다.
  4. `snaps.tmp` → `snaps.bin` 커밋이 성공한다.
  5. 프로세스는 **Exit Code 0** 으로 종료한다.

- 이때 생성된 `snaps.bin`의 바이트열(길이, 내용, CRC32)은  
  - OS/언어/컴파일러에 상관없이 **모든 올바른 v1.0 구현에서 완전히 동일**해야 한다.

12.2. **2회차 실행 (동일 워크스페이스)**

- 동일 워크스페이스에서 동일 엔트리 포인트를 다시 실행하면:

  1. 시작 시점에 `snaps.bin`이 이미 존재함을 발견한다.
  2. “최종 파일이 이미 존재하면 덮어쓰기 금지” 규칙에 의해,
     - 어떤 파일에도 쓰기를 시도하지 않고,
     - 즉시 실패 상태를 상위로 전달한다.
  3. 프로세스는 **Exit Code 1** 으로 종료한다.

12.3. **불변성 증명**

- 2회차(및 이후) 실행으로 인한 어떤 동작도  
  - 기존 `snaps.bin`의 바이트열을 단 1비트도 변경해서는 안 된다.
- 이후 별도 검증을 통해,
  - 1회차 실행 직후의 `snaps.bin`과
  - 여러 번의 재실행 이후의 `snaps.bin`이  
  **완전히 동일한 바이트열과 CRC32** 를 유지하고 있음이 증명 가능해야 한다.

12.4. **OS 간 결정론성**

- 위 12.1~12.3의 행태와 결과(바이트열, Exit Code 0/1 패턴)는  
  - Windows / Linux / macOS에서 모두 동일하게 재현되어야 한다.

### 보완점 (근본의 끝점만)

- **명세 vs 구현 문서 분리**  
  - 위 문서는 v1.0의 **공식 명세(SSOT)** 이며,  
  - 어떤 언어로, 어떤 디렉터리 구조로, 어떤 빌드 명령으로 구현하는지는 **별도 “구현/빌드 지침” 문서**에서만 다룬다.  
  - 명세 문서에 코드/의사코드를 추가하는 것은 v1.0에서 금지한다.

---

### 확장점 특정 (= 끝점 특정)

- v1.0에서 더 이상 확장하지 않는 항목:
  - Term 구조(3필드, 16바이트) 및 순서.
  - snapshot_tick = 1023 및 Term 개수 1024.
  - Header 4바이트 바이트열 `[0x4D, 0x42, 0x00, 0x01]`.
  - LCG 파라미터(M, A, C, Seed)와 tick 당 3회 호출 규칙.
  - CRC32 파라미터 및 “Header+Term 전체”를 대상으로 하는 범위.
  - Exit Code 도메인(0/1)과 “2회차 실행 → 결정론적 FAIL(1)” 패턴.
  - 단일 파일 모델 (`snaps.bin` / `snaps.tmp` 2개 이름만 사용).

이 항목들은 모두 v1.0에서 **종료선(끝점)** 으로 확정되며, 변경은 v1.1+에서만 논의할 수 있다.

### 목표점 특정  

- 현재 상태에서의 v1.0 **합격 목표**:
  - 어떤 구현이든, 위 명세만을 근거로 구현했을 때  
    - 1회차 실행: Exit Code 0, 동일한 `snaps.bin` 바이트열 생성.  
    - 2회차 실행: Exit Code 1, `snaps.bin` 불변.  
  - 이 패턴이 Windows / Linux / macOS 전체에서 동일하게 재현될 때,  
    BareMetal v1.0은 **완전히 닫힌 결정론적 시스템**으로 합격이다.
